Requirement Understanding
- We had to create an API/App which can fetch data from database deployed in another pod.
- Database should not be exposed to public.
- Database should be persistance.
- Pods interaction should not happen using pod ip.
- Pods should be self healing. 
- FinOps should be considered in design. 
- API should be accessible using loadbalancer and not Node IP

Service API Tier
- The app is built using Python + FastAPI, a widely adopted and stable framework.
- A `/health` endpoint checks database connectivity and supports Kubernetes probes.
- The root endpoint reads from `public."TestTable"` based on the requested `id`.
- Secrets is used for the database connection string.
- The app supports rolling updates strategy through a Deployment.
- The API is accessible from outside the cluster via EKS Ingress.
- Health checks are in place, so Kubernetes can detect and recover unhealthy pods.
- The service supports autoscaling through HPA.



Database Tier
- PostgreSQL runs in a StatefulSet with a single replica.
- Data is persisted using a PVC-backed volume.
- The database is only accessible inside the cluster through a headless service. Headless service provides stable internal DNS.
- The database pod uses a ConfigMap for non-secret settings like username, database name, and PGDATA.
- Sensitive data like the password is injected through a Secret.
- A liveness probe using `pg_isready` helps Kubernetes restart the database if it becomes unhealthy.
- "TestTable" has 8 records.

Kubernetes Requirements
- All resources run in the `nagp` namespace for isolation.
- The app deployment uses RollingUpdate and includes readiness/liveness probes.
- HPA scales the app based on CPU and memory usage, with a range of 1 to 4 replicas.
- The backend API is exposed externally via EKS Ingress and Loadbalancer.
- The database is not exposed outside the cluster and is persistent.
- ConfigMaps and Secrets separate configuration from pod definition.
- Secrets are used for sensitive values, including database credentials and connection string.
- Pods connect using service DNS names, not pod IPs.

Other Requirements
- Database configuration is supplied through a ConfigMap for the database pod.
- The database password is stored in a Secret(Base64), not in plain YAML.
- The persistent volume claim keeps data safe even if the database pod is restarted.
- Service discovery uses ClusterIP/headless services to avoid direct pod IP usage.

FinOps Requirements
- CPU and memory requests and limits are defined for both the application and database pods.
- HPA is configured with observed CPU and memory metrics.
- The architecture includes several cost-optimization opportunities.

Assumptions
- The application is already built and pushed to a registry accessible by the Kubernetes cluster.
- The cluster has permissions to provision storage and load balancers.
- Database initialization and seeding are handled outside the manifest files or through an additional init process.
- All app and database resources run in the `nagp` namespace.
- The `DATABASE_URL` is injected by Kubernetes Secrets into the app pod.
- The ingress controller is already installed for the target environment (ALB for EKS).
- The backend API uses query parameter ID to fetch data from database.


Justification for the Resources Used

Namespace
- Using `nagp` keeps the app environment isolated and simplifies resource management.

Deployment
- The rolling update strategy ensures deployments don’t interrupt users.
- Resource requests and limits keep pod resource usage predictable and prevent overconsumption.
- Health probes let Kubernetes restart or remove unhealthy pods automatically.

Service
- `fastapp-service` exposes the app inside the cluster so Ingress can route traffic safely.
- This avoids using pod IPs directly and keeps the architecture resilient.

Ingress
- `app_eks_ingress.yml` is for AWS ALB on EKS.

HPA
- The HorizontalPodAutoscaler proves that the app can scale automatically.
- Using both CPU and memory metrics makes scaling more accurate.
- Stabilization policies prevent rapid up/down flapping.

Database StatefulSet
- StatefulSet is the right choice for stateful workloads like databases.
- Persistent volume claims prevent data loss on rescheduling.
- The headless service provides a reliable internal address for the app.
- ConfigMaps and Secrets keep configuration separate from code and manifest templates.

Cost Optimization Justification
- Requests and limits prevent resource waste and reduce unexpected cloud spend.
- HPA allows the app to shrink to a smaller footprint when traffic is low.
- A defined storage size avoids over-provisioning the database.
- NodeGroup was t3.small with two node only
